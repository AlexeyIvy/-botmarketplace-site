# 44. Strategy Engine Overview

Статус: draft
Владелец: core trading / lab
Последнее обновление: 2026-04-23

## Цель

На данном этапе проект уже имеет рабочее ядро strategy engine:
- DSL-driven backtest;
- MTF-resolver;
- DCA execution model;
- SMC pattern primitives внутри evaluator/runtime слоя;
- Lab preview для dry-run стратегии на ограниченном окне исторических данных (до 168 часов, по умолчанию 24).

Следующая цель — не "дописать бэктест", а сделать движок более:
- переиспользуемым;
- реалистичным;
- измеримым;
- пригодным для систематической оптимизации стратегий.

## Почему нужен отдельный пакет документов

Текущая реализация функционально сильная, но часть важной логики сосредоточена в крупном `dslEvaluator.ts`, где одновременно находятся:
- вычисление базовых индикаторов;
- signal evaluation;
- side resolution;
- exit handling;
- DCA lifecycle;
- MTF resolution;
- итоговый backtest runtime.

Чтобы не создавать слишком крупные задачи для исполнителя, дальнейшая работа разбивается на несколько самостоятельных документов с маленькими шагами и узким scope.

## Основные направления

### 1. Indicator engine extraction

Сейчас публичный `apps/api/src/lib/indicators/index.ts` экспортирует только часть индикаторов. SMA, EMA, RSI и Bollinger остаются реализованными внутри `dslEvaluator.ts`, что ухудшает переиспользование и размывает архитектурные границы.

Цель направления:
- вынести базовые индикаторы в `apps/api/src/lib/indicators/`;
- сформировать единый публичный API индикаторов;
- уменьшить связность `dslEvaluator.ts`.

Отдельное решение в этом направлении:
- явно определить судьбу SMC primitives — выносятся ли они в reusable engine сейчас или остаются во второй волне refactoring.

Решение по SMC primitives принимается по следующему правилу:
- если в рамках этапа появляется более одного нового публичного потребителя SMC-логики вне `dslEvaluator.ts` / runtime-слоя, вынос выполняется в этом этапе;
- иначе SMC остаются внутренней частью evaluator/runtime и выносятся отдельной второй волной.

Подробности: `docs/45-indicator-engine-extraction-plan.md`

### 2. Backtest realism

Сейчас backtest-движок использует модель исполнения `fillAt: "CLOSE"` как единственный поддерживаемый вариант.

Цель направления:
- добавить более реалистичный execution mode;
- уменьшить риск lookahead bias;
- сделать assumptions исполнения явными, тестируемыми и документированными.

Подробности: `docs/46-backtest-realism-plan.md`

### 3. Strategy optimizer

Сейчас Lab умеет запускать single-run preview и backtest, но не покрывает массовый controlled перебор параметров DSL-конфигурации.

Цель направления:
- добавить controlled parameter sweep;
- ранжировать результаты по выбранной метрике;
- заложить основу для исследовательского workflow в Lab.

Подробности: `docs/47-strategy-optimizer-plan.md`

### 4. Walk-forward validation

Текущий backtest даёт итоговый отчёт по одному прогону, но не автоматизирует разделение in-sample и out-of-sample оценки.

Цель направления:
- внедрить walk-forward validation;
- уменьшить риск переобучения;
- сделать оценку стратегии ближе к реальной исследовательской практике.

Подробности: `docs/48-walk-forward-plan.md`

### 5. Backtest metrics expansion

Текущий `DslBacktestReport` уже подходит для базовой оценки, но недостаточен для качественного сравнения нескольких стратегий между собой.

Цель направления:
- добавить дополнительные risk-adjusted metrics;
- улучшить пригодность отчёта для ranking и optimization;
- подготовить базу для более сильной аналитики в Lab.

Подробности: `docs/49-backtest-metrics-plan.md`

## Порядок выполнения

Работы делятся на две дорожки.

### Дорожка A — основной путь
1. Backtest realism
2. Backtest metrics
3. Walk-forward validation
4. Strategy optimizer

### Дорожка B — архитектурный refactoring
1. Indicator engine extraction

`Indicator engine extraction` желательно начать как можно раньше, но он не должен жёстко блокировать realism/metrics, если для них достаточно точечных изменений без расширения публичного indicator API.

## Общие правила для всех следующих документов

Каждый дочерний документ должен:
- покрывать только одно направление работ;
- делить реализацию на задачи, каждая из которых исполняется в рамках одного небольшого PR;
- явно перечислять файлы для изменения;
- содержать тест-план;
- содержать критерии готовности;
- содержать секцию "Не входит в задачу".

## Общие инженерные ограничения

### Backward compatibility

Расширение `DslBacktestReport` должно быть:
- либо обратно совместимым;
- либо сопровождаться явным `reportVersion`.

Нельзя неявно ломать текущих потребителей отчёта.

Минимальные правила совместимости:
- новые поля отчёта добавляются только additive;
- существующие поля `DslTradeRecord` не переименовываются и не меняют тип без versioning;
- enum-поля, включая `exitReason`, расширяются только additive;
- если требуется несовместимое изменение формы отчёта, вводится `reportVersion`.

Триггер для `reportVersion` (зафиксирован в `docs/49`):

> Поле или семантика существующего поля `DslBacktestReport` или `DslTradeRecord` меняется несовместимо — переименование, изменение типа, изменение единиц измерения, изменение знака. Additive (новые опциональные поля или required-поля с `null`-семантикой) — **НЕ** триггер.

Когда триггер срабатывает, в `DslBacktestReport` вводится поле `reportVersion: 1`. Записи без поля интерпретируются как `reportVersion: 0`. До первого срабатывания триггера `reportVersion` отсутствует.

### Детерминизм

Все новые исследовательские режимы должны быть воспроизводимыми:
- одинаковый DSL;
- одинаковый набор свечей;
- одинаковые execution assumptions;
- одинаковый результат.

Если добавляется tie-breaking, sampling или ranking with equal scores, поведение должно быть deterministic.

Если добавляются sampling, random search или иные PRNG-based режимы, они обязаны использовать явно передаваемый `seed` в API или deterministic default при отсутствии `seed`.

### Scope discipline

В этот этап не входят:
- рефакторинг worker orchestration;
- изменения exchange execution layer;
- крупная переработка Lab UI beyond minimal integration;
- distributed optimization;
- infra/deploy задачи, не связанные напрямую с strategy engine;
- изменения контракта между strategy engine и worker.

## Ожидаемый результат этапа

После завершения пакета документов strategy engine должен:
- иметь более чистую архитектуру indicator layer;
- поддерживать более реалистичный режим исполнения в backtest;
- выдавать более сильные метрики качества;
- поддерживать walk-forward оценку;
- быть готовым к controlled parameter optimization в Lab.
